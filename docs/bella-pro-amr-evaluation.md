# 貝拉 Pro 擔任智慧社區配送 AMR 評估報告

**評估日期**: 2025/12/30
**專案**: 智慧社區包裹配送系統 v3.1
**評估對象**: 普渡貝拉 Pro (Bella PRO)

---

## 1. 評估摘要

| 評估結論 | 說明 |
|----------|------|
| **整體評估** | **不建議直接採用** |
| 主要原因 | 機械結構不匹配、缺乏皮帶傳送機構 |
| 替代方案 | 需採用具備皮帶傳送的工業 AMR，或重新設計對接機構 |

---

## 2. 專案 AMR 需求

根據智慧社區包裹配送系統 v3.1 規格書，AMR 需求如下：

### 2.1 機械需求

| 需求項目 | 規格 |
|----------|------|
| 承載平台 | 50cm × 45cm **雙向皮帶** |
| 承載平台高度 | **35cm**（與站點對接） |
| 載重 | ≥ 15 kg |
| 尺寸 | 55cm × 55cm × 75cm |
| 對接方式 | **皮帶傳送自動對接** |

### 2.2 導航需求

| 需求項目 | 規格 |
|----------|------|
| 導航方式 | SLAM + 二維碼輔助 |
| 定位精度 | ±3 cm |
| 避障感測 | 雷射雷達 + 超音波 |

### 2.3 功能需求

| 需求項目 | 規格 |
|----------|------|
| 電梯對接 | 自動呼叫電梯 |
| 續航時間 | ≥ 8 小時 |
| 自動充電 | 支援 |
| 雙向運送 | 大廳站 ↔ 樓層站 |

## 3. 貝拉 Pro 能力分析

### 3.1 API 能力 (符合)

| 功能 | API | 評估 |
|------|-----|------|
| 呼叫到指定點位 | 發起呼叫任務 | ✅ 支援 |
| 任務狀態追蹤 | `notifyCustomCall` 回調 | ✅ 支援 |
| 即時位置監控 | `notifyRobotPose` 回調 | ✅ 支援 |
| 主動查詢位置 | 通知機器上報位置 | ✅ 支援 |
| 地圖點位管理 | 獲取機器當前使用地圖的點位 | ✅ 支援 |
| 電量監控 | `notifyRobotPower` 回調 | ✅ 支援 |
| 充電記錄 | 充電記錄-查詢 List | ✅ 支援 |
| 異常監控 | `robotErrorNotice` 回調 | ✅ 支援 |
| 配送任務 | 配送任務 API | ✅ 支援 |

### 3.2 機械規格 (不符合)

| 項目 | 專案需求 | 貝拉 Pro | 評估 |
|------|----------|----------|------|
| 承載方式 | 雙向皮帶傳送 | **多層托盤（人工取放）** | ❌ 不符合 |
| 承載高度 | 35cm 固定高度 | **多層托盤高度不一** | ❌ 不符合 |
| 對接機構 | 自動皮帶對接 | **無自動對接** | ❌ 不符合 |
| 載重 | 15kg | 約 10-15kg（分散各層）| ⚠️ 需確認 |

### 3.3 待確認事項

| 項目 | 說明 | 重要性 |
|------|------|--------|
| 定位精度 | `notifyRobotPose` 座標精度（mm/cm/m） | 高 |
| 更新頻率 | 位置回報頻率（Hz） | 高 |
| 航向角 | 是否包含機器人朝向 | 中 |
| 電梯對接 | 是否支援電梯自動呼叫 | 高 |
| 到達判定 | 到達目標點的距離閾值 | 中 |

## 4. 關鍵差異分析

### 4.1 核心問題：機械結構不匹配

```
專案設計                              貝拉 Pro 實際結構

┌─────────────────┐                   ┌─────────────────┐
│                 │                   │   ┌─────────┐   │
│   ╔═══════════╗ │                   │   │ 托盤 3  │   │  ← 人工取放
│   ║   箱體    ║ │                   │   ├─────────┤   │
│   ╚═══════════╝ │                   │   │ 托盤 2  │   │  ← 人工取放
│                 │                   │   ├─────────┤   │
│   ═══════════   │ ← 皮帶傳送        │   │ 托盤 1  │   │  ← 人工取放
│                 │                   │   └─────────┘   │
└─────────────────┘                   └─────────────────┘

需求：自動皮帶對接                     實際：人工取放托盤
```

### 4.2 設計理念差異

| 項目 | 專案設計理念 | 貝拉 Pro 設計理念 |
|------|--------------|-------------------|
| 目標場景 | 全自動無人配送 | 半自動送餐服務 |
| 取放方式 | 站點皮帶自動傳送 | 服務員/顧客人工取放 |
| 載體 | 封閉箱體（40cm深） | 開放托盤 |
| 對接 | 機械對接（皮帶銜接）| 停靠等待人工 |

## 5. 可行性方案分析

### 5.1 方案 A：改裝貝拉 Pro（不建議）

| 項目 | 說明 |
|------|------|
| 改裝內容 | 移除托盤，加裝皮帶承載平台 |
| 優點 | 可繼續使用貝拉 Pro 底盤和 API |
| 缺點 | 破壞原廠設計、失去保固、改裝成本高、穩定性風險 |
| 評估 | ❌ 不建議 |

### 5.2 方案 B：調整專案設計（可考慮）

| 項目 | 說明 |
|------|------|
| 調整內容 | 放棄皮帶自動對接，改為人工取放 |
| 流程 | AMR 到站 → 系統通知 → 人員取放箱體 |
| 優點 | 可直接使用貝拉 Pro |
| 缺點 | **違反「全自動」設計原則**、需要人力介入、效率降低 |
| 評估 | ⚠️ 僅適用於有人值守的場景 |

### 5.3 方案 C：採用工業 AMR（建議）

| 項目 | 說明 |
|------|------|
| 方案 | 採用可定制皮帶承載的工業級 AMR |
| 優點 | 完全符合專案需求、穩定可靠 |
| 缺點 | 需重新評估 AMR 供應商、可能成本較高 |
| 候選 | 海康威視、極智嘉、快倉等工業 AMR |
| 評估 | ✅ **建議採用** |

### 5.4 方案 D：混合方案（可考慮）

| 項目 | 說明 |
|------|------|
| 方案 | 貝拉 Pro + 外掛皮帶傳送模組 |
| 設計 | 在貝拉 Pro 頂部加裝獨立皮帶傳送模組 |
| 優點 | 保留貝拉 Pro API 優勢、不破壞原機 |
| 缺點 | 增加高度和重量、需自行開發皮帶模組控制 |
| 評估 | ⚠️ 需要原型驗證 |

## 7. 結論與建議

### 7.1 結論

| 項目 | 結論 |
|------|------|
| 貝拉 Pro 能否擔任本專案 AMR | **否** |
| 主要原因 | 機械結構不匹配，缺乏皮帶自動對接 |
| API 能力 | 充足，可滿足調度需求 |
| 機械能力 | 不足，無法自動傳送箱體 |

### 7.2 建議

1. **短期**：評估其他具備皮帶傳送的 AMR 供應商
2. **中期**：若堅持使用普渡產品，洽詢普渡是否有工業級/可定制型號
3. **備選**：考慮混合方案，在貝拉 Pro 上加裝皮帶模組（需原型驗證）

### 7.3 下一步行動

| 優先級 | 行動項目 |
|--------|----------|
| P0 | 向普渡詢問是否有支援皮帶傳送的機型 |
| P0 | 評估海康/極智嘉/快倉等工業 AMR |
| P1 | 確認貝拉 Pro 的定位精度、電梯對接能力 |
| P2 | 評估混合方案的技術可行性 |
